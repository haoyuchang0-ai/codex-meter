# Codex Meter 真实任务状态与信号灯设计

日期：2026-07-10

## 目标

让悬浮窗的状态提示反映这台 Mac 上所有 Codex 任务的真实运行状态，而不是额度请求的刷新状态。状态读取必须完全在本地完成，不发送模型请求，也不消耗 Codex 额度。

正常状态包括：

- `待确认`：至少一个任务正在等待授权或用户回答。
- `工作中`：至少一个任务正在运行，且没有任务等待用户处理。
- `已完成`：所有任务已结束，并处于完成后的 8 秒提示期。
- `空闲`：没有正在运行的任务，完成提示期也已结束。

## 已确认的产品决策

- 监控范围是全局：任意 Codex 任务都会影响状态。
- 多任务聚合优先级为 `待确认 > 工作中 > 已完成 > 空闲`。
- `已完成`停留 8 秒；期间若有新任务开始，立即切换为`工作中`。
- 主窗口采用 B 方案“状态信号条”：红、黄、绿三灯常驻，只点亮当前状态对应的灯。
- 额度每 60 秒更新一次；任务状态每 1 秒在本地更新一次。
- 不增加提示音，不使用循环闪烁，避免持续干扰。

## 技术约束与结论

Codex app-server 协议提供 `active`、`waitingOnApproval`、`waitingOnUserInput` 和 `idle` 等运行状态，但桌面版 app-server 使用父子进程的私有标准输入输出管道。独立启动的第二个 app-server 读取同一任务时只会得到 `notLoaded`，不能作为桌面任务状态来源。

因此采用“本地会话记录 + 轻量 Hooks”的混合方案：

- 会话记录负责可靠识别任务开始与完成，并支持应用重启后的状态恢复。
- Hooks 负责及时识别等待授权、等待用户回答以及恢复工作。
- 两种来源只读取或写入本地状态，不调用模型。

## 组件设计

### 1. 会话记录扫描器

扫描 `~/.codex/sessions/**/rollout-*.jsonl` 中最近变更的会话文件，只解析事件类型、任务 ID 与时间：

- `task_started`：该任务进入运行态。
- `task_complete`：该任务结束。

扫描器不读取、保存或返回用户提示词、模型回复、工具参数及文件内容。首次启动只扫描近期文件，后续根据文件修改时间做增量读取，避免反复解析全部历史记录。

### 2. Hook 状态桥

仓库提供一个幂等安装脚本，将 `codex-meter-state.py` 安装到 `~/.codex/hooks/`，并在保留现有配置的前提下合并以下 Hooks：

- `UserPromptSubmit`：记录`工作中`。
- `PermissionRequest`：记录`待确认`。
- `PreToolUse` 匹配 `request_user_input`：记录`待确认`。
- `PostToolUse` 匹配 `request_user_input`：记录`工作中`。
- `Stop`：记录`已完成`。
- `SessionStart`：清理该会话遗留的过期状态。

Hook 只写入会话 ID、状态和时间戳到 `~/.codex/codex-meter/activity/`。安装前备份现有 `hooks.json`，不删除、不改写用户已有 Hook。安装完成后 README 提示重启 Codex，并在 Codex 要求时确认新增 Hook。

### 3. 状态聚合器

本地服务将会话记录与 Hook 状态合并为全局快照：

1. 任意会话处于等待态，返回`待确认`。
2. 否则，任意会话存在未完成任务，返回`工作中`。
3. 否则，最近一次完成距当前不超过 8 秒，返回`已完成`。
4. 否则返回`空闲`。

运行或等待记录超过 12 小时且没有任何后续事件时视为过期，防止 Codex 异常退出后状态永久卡住。新的开始、等待、恢复和完成事件必须覆盖旧状态。

### 4. 本地状态接口

新增 `GET /api/activity`，返回：

```json
{
  "status": "working",
  "updatedAt": "2026-07-10T10:00:00.000Z",
  "activeCount": 1,
  "waitingCount": 0,
  "source": "local"
}
```

接口不返回任务标题、提示词、目录或文件名。原有 `/api/rate-limits` 保持不变。

### 5. macOS 客户端

Swift 客户端每秒读取 `/api/activity`。额度请求只负责更新额度和 `Codex !` 错误提示，不再写入任务活动状态。

若状态接口短暂失败，保留最近一次有效状态最多 10 秒；超过 10 秒仍不可用时，显示全灯低亮和`状态未知`，避免把异常误报为空闲。恢复后立即切回真实状态。

## 视觉规格

主窗口左侧使用约 `84 x 22 pt` 的状态信号条：

- 三盏灯直径 `8 pt`，间距 `4 pt`。
- 红灯对应`待确认`，黄灯对应`工作中`，绿灯对应`已完成`。
- `空闲`时三灯均为中性灰低亮，旁边显示`空闲`。
- 活动灯使用静态轻光晕；其他灯降低饱和度和不透明度。
- 状态文字使用 `10 pt` semibold，组件尺寸固定，切换时不引发布局移动。
- 状态切换使用约 `200 ms` 的透明度与光晕过渡，并尊重 macOS“减少动态效果”设置。
- 颜色之外始终显示文字，并提供完整的 accessibility label。

胶囊模式使用同一组三灯的缩小版本，并保留状态文字和额度数字。菜单栏空间有限，继续使用单个活动色圆点与额度数字，悬停提示展示完整状态。

奶油蓝和简约仪表盘配色只改变背景与文字；红黄绿语义色在两个主题中保持一致。信号条有独立的半透明承载底色，因此不会被误认为 macOS 窗口关闭、最小化和全屏按钮。

## 错误处理

- Hooks 未安装：会话记录仍可提供`工作中`、`已完成`和`空闲`，菜单中提示状态监听未完整配置。
- 会话记录中单行 JSON 损坏：跳过该行并继续处理，不影响额度刷新。
- Hooks 状态文件损坏：忽略该文件，并从会话记录恢复。
- 本地状态接口失败：使用 10 秒宽限期，之后显示`状态未知`。
- 额度接口失败：只影响额度数值和标题错误标记，不改变任务状态。

## 测试策略

### Node.js

- 用固定 JSONL 片段测试任务开始、完成、损坏行和增量读取。
- 测试单任务与多任务的状态优先级。
- 测试等待后恢复工作、完成 8 秒窗口及 12 小时过期恢复。
- 测试 `/api/activity` 不泄露任务内容和文件路径。
- 测试 Hook 安装器可以重复运行，并保留已有 Hooks。

### Swift

- 测试活动状态 JSON 解码和状态映射。
- 测试状态接口失败的 10 秒宽限期。
- 测试额度刷新不再修改任务状态。
- 测试主窗口与胶囊模式切换时组件尺寸不变化。

### 视觉与端到端验证

- 在奶油蓝与简约配色下分别检查四种状态和异常状态。
- 截图检查主窗口、胶囊模式和菜单栏，确认无重叠、无文字截断、标题仍居中。
- 实际启动一个 Codex 任务，验证`工作中 -> 待确认 -> 工作中 -> 已完成 -> 空闲`完整链路。
- 验证状态监听期间额度没有额外消耗。

## 非目标

- 不显示任务标题、提示词、目录或具体工作内容。
- 不控制、暂停或终止 Codex 任务。
- 不使用私有进程注入、标准输入输出劫持或辅助功能 UI 抓取。
- 不增加云端服务、账号登录或遥测。
